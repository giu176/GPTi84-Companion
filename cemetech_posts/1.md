## The Code:

Originally I started writing my firmware in C because that's just the embedded muscle memory for hardware projects. And with the ESP32 iteration, I figured it was going to be trivial via ESP-IDF. Then, yknow, I broke the poor thing. So I had to re-evaluate for the pivot to Pico. 

Now, I could have chosen C again here, but honestly? I had never used MicroPython and kinda wanted an excuse to see what "embedded JIT" looks like hands-on (*hint: highly constrained but usable and fun!*)

So that's what I settled on, and I got to work figuring out what on earth I was actually going to do here to make any of this vision pan out the way I imagined.

I started simple. Needed access to a simple set of things: the pins coming out of that 2.5mm calculator I/O port and into that orange jack I had "donated" from the other PCB.

The way I had wired things was the Tip of the link cable going into GPIO6 and the Ring going into GPIO7 (worth noting: cable used was TRRS but it changes nothing since the second Ring just shorts to ground anyways).

Got those wired up as constants in my firmware and got to work... basically throwing stuff at the wall to map the surface area. Started with the hardware basics:

```py
TIP  = 6 # 0-line
RING = 7 # 1-line

# Start as inputs with pullups (released, pullup wins).
# Pico has no external pullups on these pins, so without Pin.PULL_UP
# the line floats while we are not driving and only the calc-side pullup
# pulls it high; that rise is slow enough to glitch the calc's bit reader
tip  = Pin(TIP,  Pin.IN, Pin.PULL_UP)
ring = Pin(RING, Pin.IN, Pin.PULL_UP)

def release(p):
    p.init(mode=Pin.IN, pull=Pin.PULL_UP)

def pull_low(p):
    p.init(mode=Pin.OUT, value=0)

def idle():
    release(tip); release(ring)

def read():
    # returns (tip_level, ring_level), both 1 = idle
    return tip.value(), ring.value()
```

Idle high, pulled low to talk, ack on the other line: nothing surprising if you've already squinted at the link guide. The thing I didn't appreciate until I was actually sitting there poking at it was how unforgiving the timing is when your "send" function is interpreted Python on a chip that's also doing GC behind your back. The first version of `send_bit` worked beautifully into a logic analyzer and got completely ignored by the calculator. The fix was:

stop being clever, hold the line until the receiver acks, release, wait for the ack to clear, move on.

Once I stopped trying to pace it and just let the two sides clock each other, MicroPython was more than fast enough for my needs.

So with bits going across, bytes are just eight of those LSB-first and a packet is `[machine][cmd][len_lo][len_hi]` followed by an optional body and a 16-bit sum.
I won't labor the framing, the [link guide](https://www.ticalc.org/archives/files/fileinfo/247/24750.html) has it.

The one thing worth flagging for anyone else doing this from scratch is that a handful of the silent-link commands (CTS, ACK, ERR, RDY, EOT, VER) are header-only on the wire even when their length field is nonzero. If you trust the length field you sit there forever waiting for body bytes that are never coming. Ask me how I know, haha.

The first real milestone was getting the calc to respond to anything at all. I picked `RDY` (0x68) as the hello-world because it's the cheapest possible round trip: PC sends `[0x23 0x68 0x00 0x00]`, a listening calc replies with an `ACK`, and that's it.

No body, no checksum, no variable headers, just "are you there" / "yes".

The first time I saw the ACK actually come back was cathartic!

From there it was a fairly mechanical climb up the protocol.
`VER` to pull OS and hardware version bytes back (sanity check that I was framing correctly and not just getting lucky on a one-byte command). Then the variable-transfer dance: the PC sends a `VAR` header describing what's about to land, the calc replies `CTS` if it has room, the PC sends a `DATA` packet with the actual body, calc `ACK`s, PC sends `EOT`.

Four packets to move one variable, and every one of them has to be byte-perfect or the calc just sits there and you have no idea which side dropped the ball.

The thing that unstuck me on the variable transfer was realizing the machine ID isn't the *partner*, it's the *sender*. Both sides stamp their own ID on every packet they put on the wire. So when the calc initiates and sends 0x73, the PC's reply isn't also 0x73, it's 0x23. There's a whole little table of these pairs:

```py
def pc_id_for(calc_id):
    pairs = {0x82: 0x02, 0x83: 0x03, 0x73: 0x23,
             0x86: 0x06, 0x88: 0x08, 0x98: 0x88}
    return pairs.get(calc_id, 0x23)
```

Once that clicked, the rest of it fell into place pretty fast.

The proof-of-life I held myself to was simple. Data across the line, bidirectionally. {4, 2, 0} was, for certain reasons, the packet chosen here... but there were constraints. No tilem, no TI Connect, no CALCnet. MicroPython on the Pico talking the raw protocol to the OS's idle silent-link receiver.

![Calc home screen showing three Send({4,2,0}) Done lines, the proof-of-life round trip](https://cdn.xandwr.com/cemetech/post-1/06_send-420-done.png)

That worked. And then, because I didn't trust it, I locked a 2.4KB program, pushed it across, dumped it back, and diffed the bytes. Identical.

At that point I called the wire layer done and started actually thinking about the fun part: what do you put on top of a working bit-banged DBUS stack when you've got a Pico W with wifi sitting one trace away from the link port?

I'll talk about this part briefly, since it actually has a couple of cute bits to it.

The 84+ gives you these built-in string variables you can read and write from any program: Str1, Str2, all the way through Str9, then Str0. I figured I'd just reuse those as the prompt format instead of inventing a custom one.

So Str1 is the prose part of your prompt ("what colour is the sky") and Str2 is the math part ("2X+3"). The reason for the split is that the desktop side detokenizes them differently: text mode keeps "HELLO" as a word, math mode turns "2X" into "2*X" so the model actually sees multiplication. The user is the one telling us which is which by putting it in the right slot, which is way easier than trying to guess on the wire.

The reply comes back paginated across Str3 through Str9 and then Str0: up to 8 pages of 112 chars each, which is exactly the 16x7 large-font screen. Then real variable N gets pushed last with the page count. That's the bit I'm most happy with: pushing N last means the BASIC side either sees N=0 ("WAITING...") or it sees the real count and all the pages are already sitting in their slots. No partial state to worry about.

The whole UX on the calc is then just a tiny BASIC program that prompts you for Str1 and Str2, calls the asm bridge, sits on `Pause N` until the reply lands, and then arrow-keys you through the pages. The asm stays a dumb pipe: send the two strings, exit, let the calc go idle so the OS's silent-link receiver can accept the reply.
